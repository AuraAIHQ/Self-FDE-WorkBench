> ⚠️ 说明：本轮客户仅给出一句话「帮小区做一个团购小工具」。以下多为基于社区团购行业惯例的**初稿假设**，标【假设】者需客户确认（见 GAPS.md）。

## 1. 问题定义
小区里的团购目前多依赖微信群手动接龙、Excel 统计，存在痛点【假设】：
- 团长手动汇总订单、算钱、对账，易错且费时。
- 居民下单要翻聊天记录，不知道成团进度、截止时间、自提信息。
- 缺少历史订单、统一的商品与价格展示。

需要一个轻量 Web 工具，让**团长一键发起团购、居民一键参团、系统自动汇总订单与金额**。

## 2. 目标
- 团长发起一次团购的操作 < 2 分钟。
- 居民下单流程 ≤ 3 步（看商品 → 填数量/信息 → 提交）。
- 团长可一键导出/查看某次团购的全部订单与应收金额，无需人工统计。
- 无需安装 App，微信内打开网页即用【假设：主要在微信内使用】。

## 3. 范围（MVP，M0）
- 团购活动的发起、编辑、关闭（团长）。
- 商品条目：名称、单价、单位、图片、限购/库存（可选）。
- 居民浏览进行中的团购并下单（选商品、数量、留联系方式/自提点）。
- 订单汇总：团长查看每单明细、总数量、应收总额、按商品统计。
- 成团进度展示（已下单人数/份数、距截止时间）。

## 4. 范围外 / 非目标（MVP 阶段，待确认）
- 【假设】不做线上支付资金托管（先做「记录订单+线下收款」，线上支付列为 M1 待确认）。
- 不做物流配送体系（默认小区自提）。
- 不做供应链/进货/库存中台。
- 不做复杂营销（拼团裂变、优惠券、分销佣金）——M2 再议。
- 不做多小区 SaaS 多租户运营后台（先服务单个小区）。

## 5. 成功指标（待客户确认口径）
- 首个真实团购活动能全程跑通：发起 → ≥N 位居民下单 → 团长成功汇总收款。
- 团长汇总统计零手工 Excel。

## 6. 关键未决（详见 GAPS.md）
1. 运营/团长角色模型：物业统一运营，还是任意居民可发起？
2. 是否需要线上收款（微信支付），还是仅记录订单、线下收钱？
3. 登录与身份：微信授权登录 / 手机号 / 免登录填信息？
4. 自提 vs 配送、核销方式。
