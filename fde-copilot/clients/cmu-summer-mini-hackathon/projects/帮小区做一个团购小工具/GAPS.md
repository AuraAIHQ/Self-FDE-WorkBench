# 缺口台账 · Gaps

> 客户：CMU Summer Mini Hackathon ｜ 项目：帮小区做一个团购小工具

## 待客户回答
> 均为「只有客户能拍板」的商业/范围决策，决定后可显著推进 loop-ready。

- **Q1 团购模式**：是「单团长制」（一个热心邻居/物业发起并收单）还是「人人可发起」（任意居民都能建团）？默认按**单团长制**做。
- **Q2 是否要在线支付**：本次是仅登记订单+线下收款，还是需要在线付款（微信/Stripe）？默认**免支付、线下收款**（黑客松更易 demo）。若要在线支付将显著增加范围。
- **Q3 居民身份/校验强度**：居民下单是否免注册、仅填姓名+手机+门牌即可？手机号是否需要短信验证码校验真实性？默认**免注册 + 不强制验证码**（手机号非空即可）。
- **Q4 重复下单处理**：同一居民再次下单，是「追加为新订单」还是「按手机号合并/覆盖」？默认**追加为新订单**。
- **Q5 团长管理页授权方式**：用创建时生成的隐藏管理链接（adminToken）即可，还是需要给团长设登录口令？默认**adminToken 链接授权**（够 demo 用）。
- **Q6 部署诉求**：黑客松 demo 只需本地/临时公网链接跑通，还是要交付一个持续可用的线上服务？默认**临时公网 demo 链接（Vercel）**。
- **Q7 品牌/命名/语言**：工具叫什么名字？界面语言中文即可吗？默认**中文、暂用占位名"邻里团"**。

## 调研假设·待确认
> 以下由 FDE 基于社区团购通用领域知识给出，属**假设**，需客户确认。

- 【调研假设·待确认】采用经典「社区团购团长制」链路：发起团购 → 居民下单 → 截单汇总 → 线下自提。来源：社区团购通用业务模型（团长制）。
- 【调研假设·待确认】黑客松场景下移动端优先、免支付登记式最易在有限时间内 demo 出完整闭环。来源：FDE 经验判断。
- 【调研假设·待确认】技术选型 Next.js + Prisma + SQLite/Postgres + Tailwind，部署 Vercel（技术实现细节，FDE 拍板，客户如无异议即采用）。
