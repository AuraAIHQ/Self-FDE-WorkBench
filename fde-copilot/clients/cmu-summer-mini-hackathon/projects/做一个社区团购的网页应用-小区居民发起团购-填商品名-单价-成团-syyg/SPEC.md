## 一句话定位
小区团购发起与参团管理页面 —— 居民快速发起团购、一键参团、实时展示进度、自动统计订单，全程本地存储、零支付流程。

## 目标用户
小区居民 + 志愿者（非技术背景，需要极简UI）

## 核心功能
| 功能 | 验收标准 |
|------|--------|
| 发起团购 | 输入商品名/单价/成团人数/截止时间，点击"发起"后显示在页面上 |
| 参团 | 选中团购→一键点击"我要参团"→显示参团确认 |
| 实时进度展示 | 页面显示"已参xx人/需xx人"文字 + 进度条 |
| 自动统计 | 达成团人数或时间截止时，自动生成订单清单（显示参团者、商品名、单价、总数、总额） |
| 数据持久化 | 使用 localStorage 存储团购列表和参团记录，页面刷新不丢失 |

## 范围
### 范围内
- 单页面应用（HTML + 纯 JavaScript）
- 团购信息的基础操作（新增、删除、列表）
- 参团记录的收集与统计
- 本地时间检测与截止判定
- 进度条/进度文字实时更新
- 订单清单生成与展示
- localStorage 数据持久化

### 范围外
- 在线支付 / 收款功能
- 用户认证 / 登录系统
- 多页面路由
- 移动端响应式设计（主要面向桌面 / 投影演示）
- 数据备份 / 云端同步
- 消息通知 / 推送

## 技术方向
- **前端框架**：纯 HTML + Vanilla JavaScript（不依赖 React / Vue）
- **存储**：Browser localStorage（无后端）
- **架构**：单页面应用（SPA）
- **时间检测**【假设·待确认】：页面加载 + 每次用户交互时检测；或改为定时器每30秒检测一次？
- **订单导出**【假设·待确认】：仅展示在页面，或支持复制文本 / 打印功能？

## 待确认 / 缺口
1. **参团身份**：参团时是否需要输入参与者的名字/电话？直接"+1"计数，还是记录个人信息？（决定表单复杂度）
2. **重复参团**：同一个人是否可以多次参加同一个团购？（允许 = 数量累加；禁止 = 需简单去重机制）
3. **截止时间检测**：是用户手动刷新页面来检测截止，还是需要自动定时轮询？
4. **订单清单内容**：仅展示"参团人数/商品/价格"，或需要记录每位参与者的详细名单？
5. **数据清理**：已成团 / 已截止的订单是否需要保留、归档、或手动删除？

## 成功指标（Hackathon验收）

- ✅ 数据不会因页面刷新而丢失
- ✅ 界面简洁，小区志愿者能在30秒内学会使用
- ✅ 演示时流畅运行、无明显卡顿
