# 需求规格 · Spec

> 客户：CMU Summer Mini Hackathon ｜ 项目：小区团购小工具（hack5）｜ 交付物：app（web）

## 一句话定位
帮助小区住户快速发起和参加团购，汇总订单清单、降低采购成本。

## 目标用户
- 小区住户（参加者）
- 小区物业/团购发起人（可选：核心人物）

## 核心功能
- **发起团购**：发起人选择商品清单、设定截止时间、邀请参加 → 验收：2分钟内完成发起
- **浏览/加入**：住户查看在线团购列表、选择商品数量、一键加入 → 验收：参加流程 <1分钟
- **订单汇总**：实时展示参加人数、各商品总数、成员清单、订单详情 → 验收：页面实时刷新
- **团购管理**：发起人编辑/结束团购；支持团购历史查看

## 范围
**范围内**：
- 浏览器 web 应用（PC/平板友好）
- 用户注册/登录（邮箱或简易身份校验）
- 团购 CRUD（创建、编辑、删除、查看）
- 实时数据同步（WebSocket 或轮询）
- 简易权限管理（谁能发起、谁能加入）

**范围外** ❌：
- 支付/扣款集成（暂定：单纯汇总，不涉及钱款）
- 配送/物流管理
- 商品库 & 库存管理（每次手工输入商品）
- 原生移动应用
- 发票/报表

## 技术方向【假设·待确认】
- **前端**：React 18 + Tailwind CSS（或 Vue 3）
- **后端**：Node.js/Express + SQLite（或 Supabase 无服务）
- **实时通知**：轮询或 WebSocket
- **认证**：邮箱注册 + Session / JWT
- **部署**：Vercel（前端）+ Railway 或本地服务器（后端）

## 待确认 / 缺口

### ⚠️ 必须客户回答的问题
| ID | 问题 | 为什么卡 |
|---|---|---|
| Q1 | **谁能发起团购？** 只有物业/管理员，还是任何住户都能发起？ | 决定权限模型、界面分角色设计 |
| Q2 | **团购什么？** 固定品类清单（如生鲜/日用品），还是自由输入商品？ | 决定数据结构和商品选择 UI |
| Q3 | **规模**：预期多少住户、多少个小区、单个团购最多多少人？ | 决定系统架构（SQLite vs 云DB）|
| Q4 | **支付怎么处理？** 团购单纯汇总，还是涉及金额计算、分账、收款？ | 涉及金额会 10 倍增加复杂度 ⚠️ |
| Q5 | **商品交付方式？** 配送、自提、还是线下物业交接？ | 订单流程设计（是否需要地址等） |
| Q6 | **已有供应商吗？** 是对接商家还是住户间互助？ | 决定是否需要卖方账号、定价逻辑 |

### 技术假设（待确认）
- 轮询（3-5s 更新一次）足够吗，还是必须 WebSocket？
- 移动响应式必需，还是 PC 优先即可？
- 用户数预期 <1000 人吗？

### 已知风险
- **黑客松时间紧**（通常 24-48h）→ 需要极度 MVP 化；支付/分账功能会打爆时间预算 ⚠️
- 若涉及金额，需额外考虑安全/审计
